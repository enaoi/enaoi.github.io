# React Hooks

React Hooks 要解决的是逻辑状态复用的问题。是继 render-props 和 higher-order components 之后的第三种状态共享方案。

# Why Hooks

⭐️ Class 组件的痛点

  1. 复用状态逻辑很难：不同的组件和生命周期之间无法复用状态逻辑
  2. 复杂组件难以理解：逻辑散布在多个生命周期内
  3. class 的缺陷： this的指向问题。

⭐️ Hooks 的优点

  1. 更加的扁平化，多个状态不会产生嵌套
  2. Hooks 可以引用其他 Hooks
  3. 更容易将 UI 和状态分离

# Hooks 规则

  1. 只能在函数最外层调用 Hook。不要在循环、条件判断或者子函数中调用。
  2. 只能在 React 的函数组件中调用 Hook。不要在其他 JavaScript 函数中调用。（还有就是自定义的 Hook 中）

# useState

React 的 state 采用 Object.is 比较算法。

  • 当前的state依赖前一个state时，必须使用函数方式
  • useState 不会自动合并并更新对象。需要手动调用展开运算符或者是 Object.assign 方法。

惰性初始 state :

initialState 参数只会在组件的初始渲染中起作用，后续渲染时会被忽略。如果初始 state 需要通过复杂计算获得，则可以传入一个函数，在函数中计算并返回初始的 state，此函数只在初始渲染时被调用：

const [state, setState] = useState(() => {
  // 复杂计算
  const initialState = someExpensiveComputation(props);
  return initialState;
});

# useEffect

useEffect完全指南

  • useEffect() 一般都执行副作用操作（执行数据获取、订阅或者手动修改 DOM 等称为“副作用”）

  • useEffect() 是在组件内声明的，所以它们可以访问到组件的 props 和 state。

# effect 的执行时机

默认情况下，React 会在每次渲染后调用useEffect()（包括第一次渲染的时候）

  • 与 componentDidMount、componentDidUpdate  不同的是，在浏览器完成布局与绘制之后，传给 useEffect 的函数会延迟调用。这使得它适用于许多常见的副作用场景，比如设置订阅和事件处理等情况，因此不应在函数中执行阻塞浏览器更新屏幕的操作。

  • 但并非所有 effect 都可以被延迟执行。例如，在浏览器执行下一次绘制前，用户可见的 DOM 变更就必须同步执行，这样用户才不会感觉到视觉上的不一致。React 为此提供了一个额外的 useLayoutEffect Hook 来处理这类 effect。它和 useEffect 的结构相同，区别只是调用时机不同。

# effect 的清除机制

effect的清除机制就是返回一个 cleanup 清除函数。effect 的清除阶段在每次渲染时都会执行，而不是只在卸载组件时执行一次。因为：

  • 因为 props 和 state 发生变化时，要及时更新清除函数内的props和state。也就是componentDidUpdate的处理。
  • useEffect 在重新调用之前会清除上一个 useEffect 已避免内存泄露 和 bug。（清除的时错误的副作用等）

# effect 的性能优化

每次渲染都需要执行 effect，我们可以对此进行优化。就是传入一个依赖数组。

  • []：不依赖任何参数，所以只执行一次。（注意：内部一直都是初始值）
  • [v1,v2,...]：任何一个依赖发生变化时，执行 useEffect。这里的比较原则是？
    • 函数能否作为依赖
      • 最好把函数移到 effect 内部，而不是做为依赖
      • 无法移动到 effec t内部的函数，至少包裹 useCallback
      • 如果是纯函数，则应该直接放在组件外部
    • 数组频繁变化
      • state 依赖自己的 prevState ： 使用 setState 的函数形式，而不能引用当前的 state 。
      • 一个 state 依赖于另一个 state ：使用 useReducer
      • 不得不使用this的功能：将 props 保存在 Ref 中

# useMemo

返回一个 memoized 函数，在某个依赖项改变时才重新计算。

传入 useMemo 的函数会在渲染期间执行。请不要在这个函数内部执行与渲染无关的操作，诸如副作用这类的操作属于 useEffect 的适用范畴，而不是 useMemo。

const memoizedValue = useMemo(() => computeExpensiveValue(a, b), [a, b]);

# useCallback

返回一个 memoized 函数。当你把回调函数传递给经过优化的并使用【引用相等性】去避免非必要渲染的子组件时，它将非常有用。

useCallback(fn, deps) 相当于 useMemo(() => fn, deps)

注意： 正常的函数会在组件第二次渲染的时候，会被垃圾收集，但是使用 useCallback 后，原来的方法不会被回收，并会创建一个新的函数，所有要权衡优化的合理性。

# useReducer

使用场景：

  1. state逻辑复杂，包含多个子值等
  2. 下一个state依赖于之前的state

优点：

  1. 对于深更新的组件做性能优化，因为可以向子组件传递 dispatch 而不是毁掉函数。
import React, { useReducer } from "react";

const initialState = {count: 0}

function reducer(state, action) {
    switch(action.type){
        case 'increment':
            return {count: state.count + 1}
        case 'decrement':
            return {count: state.count - 1}
        default:
            throw new Error('type is error')
    }
}

function Counter() {
    const [state, dispatch] = useReducer(reducer,initialState);
    return (
        <p>
            一个使用 useReducer 的 counter 组件
            Count:{state.count}
            <button onClick={() => dispatch({type:'increment'})}>+</button>
            <button onClick={() => dispatch({type:'decrement'})}>-</button>
        </p>
    )
}

export default Counter;

# useMemo 和 useCallback 使用指南

参考文档(英)

useMemo 和 useCallback 在很多情况下并不能带来性能的优化，因为优化也是要成本的。这两个 hooks 内置于 React 都有特别的原因：

  1. 引用相等：useCallback
  2. 昂贵的计算：useMemo

# 引用相等

const Foo = ({ bar, baz }) => {
  const options = { bar, baz };
  React.useEffect(() => {
    console.log('render...')
    console.log(JSON.stringify(options))
  }, [options])
  return <div>foobar</div>
}

useEffect 将对每次渲染中对 options 进行【引用相等性检查】, 由于JavaScript的工作方式，每次渲染 options 都是新的，所以当React测试 options 是否在渲染之间发生变化时，它将始终计算为 true，意味着每次渲染后都会调用 useEffect 回调，而不是仅在 bar 和 baz 更改时调用。

解决方案： 1.

function Foo({bar, baz}) {
  React.useEffect(() => {
    const options = {bar, baz}
    buzz(options)
  }, [bar, baz]) // we want this to re-run if bar or baz change
  return <div>foobar</div>}

但如果 bar 和 baz 不是原始类型，还是会报错。 2.

function Blub() {
  const bar = React.useCallback(() => {}, [])
  const baz = React.useMemo(() => [1, 2, 3], [])
  return <Foo bar={bar} baz={baz} />
  }
// 同上

优化不必要的重新渲染:

// react.memo  避免不必要的渲染，props改变时才渲染
const CountButton = React.memo(function CountButton({ onClick, count }) {
  return <button onClick={onClick}>{count}</button>
})

function DualCounter() {
  const [count1, setCount1] = React.useState(0)
  // onClick 参数是一个事件函数，必须要用callback缓存才能不会每次刷新
  const increment1 = useCallback(() => setCount1(c => c + 1), [])
  const [count2, setCount2] = React.useState(0)
  const increment2 = useCallback(() => setCount2(c => c + 1), [])
  return (
    <React.Fragment>
      <CountButton count={count1} onClick={increment1} />
      <CountButton count={count2} onClick={increment2} />
    </React.Fragment>
  )
}

# 昂贵的计算

这是 useMemo 内置于 React 的另一个原因（注意这个不适用于 useCallback）。useMemo 的好处是你可以采用如下值：

const a = {b: props.b}
// 惰性获取
const a = React.useMemo(() => ({b: props.b}), [props.b])

计算成本很高的同步计算函数 iterations 和 multiplier

function RenderPrimes({iterations, multiplier}) {
  const primes = React.useMemo(() => calculatePrimes(iterations, multiplier), [
    iterations,
    multiplier,
  ])
  return <div>Primes! {primes}</div>
}

即使你在每次渲染时定义了计算素数的函数（非常快），React只在需要值时才调用该函数。 除此之外，React还会在给定输入的情况下存储先前的值，并在给定跟之前相同输入的情况下返回先前的值。 这是 memoization 在起作用。
