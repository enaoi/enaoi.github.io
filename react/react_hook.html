# React Hooks

React Hooks 要解决的是逻辑状态复用的问题。

是继 render-props 和 higher-order components 之后的第三种状态共享方案，不会产生 JSX 嵌套地狱问题

# Why Hooks

# Class Component 的痛点

  1. 复用状态逻辑很难：不同的组件和生命周期之间无法复用状态逻辑
  2. 复杂组件难以理解：逻辑散布在多个生命周期内
  3. class 的缺陷： this的指向问题。

复用状态逻辑很难：

在不同的组件和生命周期之间无法复用状态逻辑。而由 providers，consumers，高阶组件，render props 等其他抽象层组成的组件会形成“嵌套地狱”。

render() {
  return (
    <FooContext.Provider value={this.state.foo}>
      <BarContext.Provider value={this.state.bar}>
        <BarContext.Consumer>
          {bar => (
            <FooContext.Consumer>
              {foo => ()}
            </FooContext.Consumer>
          )}
        </BarContext.Consumer>
      </BarContext.Provider>
    </FooContext.Provider>
  )
}

复杂组件难以理解：

每个生命周期常常包含一些不相关的逻辑。例如，组件常常在 componentDidMount 和 componentDidUpdate 中获取数据。但是，同一个 componentDidMount 中可能也包含很多其它的逻辑，如设置事件监听，而之后需在 componentWillUnmount 中清除。也让复杂组件变得难以理解。

class 缺陷：

  • this 的指向问题。
  • 函数组件和 class 组件的差异存在分歧。react是如何识别纯函数组件和类组件的？

# Hooks 的特性

Hook 是一个特殊的函数，它可以让你“钩入” React state 及生命周期等。不需要使用高阶函数包裹你的组件，也不需要忍受mixin的痛苦。

  1. 更加的扁平化，多个状态不会产生嵌套
  2. Hooks 可以引用其他 Hooks
  3. 更容易将 UI 和状态分离

# Hooks API

# Hooks 规则

  • 只能在函数最外层调用 Hook。不要在循环、条件判断或者子函数中调用。
  • 只能在 React 的函数组件中调用 Hook。不要在其他 JavaScript 函数中调用。（还有就是自定义的 Hook 中）

# the State Hook

React 的 state 采用 Object.is 比较算法

  1. 定义一个 state 变量
  2. setState 为这个变量的更新函数
  3. useState 方法里面唯一的参数就是变量初始值

注意：useState 不会自动合并并更新对象。需要手动调用展开运算符或者是 Object.assign 方法

const [state, setState] = useState(initialState);

setState(prevState => {
    return {...prevState, ...updateValues}
})

# 惰性初始 state

initialState 参数只会在组件的初始渲染中起作用，后续渲染时会被忽略。如果初始 state 需要通过复杂计算获得，则可以传入一个函数，在函数中计算并返回初始的 state，此函数只在初始渲染时被调用：

const [state, setState] = useState(() => {
  // 复杂计算
  const initialState = someExpensiveComputation(props);
  return initialState;
});

# the Effect Hook

解决 class 中生命周期函数经常包含不相关的逻辑，但又把相关逻辑分离到了几个不同方法中的问题。

在 React 组件中执行数据获取、订阅或者手动修改过 DOM。我们统一把这些操作称为“副作用”。useEffect 告诉 React 组件需要在渲染后执行这些副作用操作。

副作用函数是在组件内声明的，所以它们可以访问到组件的 props 和 state。默认情况下，React 会在每次渲染后调用副作用函数 —— 包括第一次渲染的时候。

useEffect() 等同于 componentDidMount、componentDidUpdate、componentWillUnmount 这三个函数的组合。

# 无需清除的 effect

useEffect((status) => {
    Api.subscribe(props.id, count)
})

React 会在组件卸载的时候执行清除操作

如果组件多次渲染，则在执行下一个 effect 之前，上一个 effect 就已被清除。

useEffect((status) => {
    Api.subscribe(props.id, count)
    //  清除函数
    return () => {
      Api.unsubscribe(props.id, count)

# effect 的执行时机

useEffect完全指南

与 componentDidMount、componentDidUpdate  不同的是，在浏览器完成布局与绘制之后，传给 useEffect 的函数会延迟调用。这使得它适用于许多常见的副作用场景，比如设置订阅和事件处理等情况，因此不应在函数中执行阻塞浏览器更新屏幕的操作。

然而，并非所有 effect 都可以被延迟执行。例如，在浏览器执行下一次绘制前，用户可见的 DOM 变更就必须同步执行，这样用户才不会感觉到视觉上的不一致。React 为此提供了一个额外的 useLayoutEffect Hook 来处理这类 effect。它和 useEffect 的结构相同，区别只是调用时机不同。

虽然 useEffect 会在浏览器绘制后延迟执行，但会保证在任何新的渲染前执行。React 将在组件更新前刷新上一轮渲染的 effect。

# effect 的条件执行

只在传入的参数发生变化时更新，可以优化性能，不必每次执行。

  useEffect(() => {
    document.title = `clied ${count} times.`
  }, [count])

# useMemo

返回一个 memoized 函数，在某个依赖项改变时才重新计算。

传入 useMemo 的函数会在渲染期间执行。请不要在这个函数内部执行与渲染无关的操作，诸如副作用这类的操作属于 useEffect 的适用范畴，而不是 useMemo。

const memoizedValue = useMemo(() => computeExpensiveValue(a, b), [a, b]);
1

# useCallback

返回一个 memoized 函数。当你把回调函数传递给经过优化的并使用【引用相等性】去避免非必要渲染的子组件时，它将非常有用。

useCallback(fn, deps) 相当于 useMemo(() => fn, deps)

注意： 正常的函数会在组件第二次渲染的时候，会被垃圾收集，但是使用 useCallback 后，原来的方法不会被回收，并会创建一个新的函数，所有要权衡优化的合理性。

# useReducer

使用场景：

  • state逻辑复杂，包含多个子值等
  • 下一个state依赖于之前的state

优点：

  1. 对于深更新的组件做性能优化，因为可以向子组件传递 dispatch 而不是毁掉函数。
import React, { useReducer } from "react";

const initialState = {count: 0}

function reducer(state, action) {
    switch(action.type){
        case 'increment':
            return {count: state.count + 1}
        case 'decrement':
            return {count: state.count - 1}
        default:
            throw new Error('type is error')
    }
}

function Counter() {
    const [state, dispatch] = useReducer(reducer,initialState);
    return (
        <p>
            一个使用 useReducer 的 counter 组件
            Count:{state.count}
            <button onClick={() => dispatch({type:'increment'})}>+</button>
            <button onClick={() => dispatch({type:'decrement'})}>-</button>
        </p>
    )
}

export default Counter;

# useMemo 和 useCallback 使用指南

参考文档(英)

useMemo 和 useCallback 在很多情况下并不能带来性能的优化，因为优化也是要成本的。这两个 hooks 内置于 React 都有特别的原因：

  1. 引用相等：useCallback
  2. 昂贵的计算：useMemo

# 引用相等

const Foo = ({ bar, baz }) => {
  const options = { bar, baz };
  React.useEffect(() => {
    console.log('render...')
    console.log(JSON.stringify(options))
  }, [options])
  return <div>foobar</div>
}

useEffect 将对每次渲染中对 options 进行【引用相等性检查】, 由于JavaScript的工作方式，每次渲染 options 都是新的，所以当React测试 options 是否在渲染之间发生变化时，它将始终计算为 true，意味着每次渲染后都会调用 useEffect 回调，而不是仅在 bar 和 baz 更改时调用。

解决方案： 1.

function Foo({bar, baz}) {
  React.useEffect(() => {
    const options = {bar, baz}
    buzz(options)
  }, [bar, baz]) // we want this to re-run if bar or baz change
  return <div>foobar</div>}

// react.memo  避免不必要的渲染，props改变时才渲染
const CountButton = React.memo(function CountButton({ onClick, count }) {
  return <button onClick={onClick}>{count}</button>
})

function DualCounter() {
  const [count1, setCount1] = React.useState(0)
  // onClick 参数是一个事件函数，必须要用callback缓存才能不会每次刷新
  const increment1 = useCallback(() => setCount1(c => c + 1), [])
  const [count2, setCount2] = React.useState(0)
  const increment2 = useCallback(() => setCount2(c => c + 1), [])
  return (
    <React.Fragment>
      <CountButton count={count1} onClick={increment1} />
      <CountButton count={count2} onClick={increment2} />
    </React.Fragment>
  )
}

# 昂贵的计算

这是 useMemo 内置于 React 的另一个原因（注意这个不适用于 useCallback）。useMemo 的好处是你可以采用如下值：

const a = {b: props.b}
// 惰性获取
const a = React.useMemo(() => ({b: props.b}), [props.b])

function RenderPrimes({iterations, multiplier}) {
  const primes = React.useMemo(() => calculatePrimes(iterations, multiplier), [
    iterations,
    multiplier,
  ])
  return <div>Primes! {primes}</div>
}
